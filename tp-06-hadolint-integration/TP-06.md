# TP-06: Integrating Hadolint in CI

**Difficulty**: Intermediate 🟡  
**Estimated Duration**: 20 min  

## Learning Objectives
- Automate Dockerfile linting using a bash script
- Simulate a Continuous Integration (CI) pipeline step.
- Ensure only valid Dockerfiles allow the pipeline to pass.

## Prerequisites
- Completed TP-05

---

## Instructions

In a real environment (GitLab CI, GitHub Actions, Jenkins), we cannot manually type `hadolint Dockerfile` every time. We automate it.

### Step 1: Examine the CI Script
Open `ci-hadolint.sh`. It is a simple bash script that searches for any file named `Dockerfile` in the current directory and subdirectories, and passes it to `hadolint`.

If the linter finds an error, the script exits with a non-zero exit code (`exit 1`), which causes a CI pipeline job to fail!

### Step 2: Run the automated check
Ensure the script is executable:
```bash
chmod +x ci-hadolint.sh
```

Run it:
```bash
./ci-hadolint.sh
```

Because the `Dockerfile` in this directory is currently "good", the script should exit successfully (Code 0).

### Step 3: Break the Dockerfile
Introduce an error into the `Dockerfile` by adding `FROM node:latest` instead of a pinned version.

Run the script again:
```bash
./ci-hadolint.sh
```
Notice how it immediately catches the rule violation and stops the script.

### Step 4: Example in GitHub Actions or GitLab CI
In **GitHub Actions**, this would look like:

```yaml
jobs:
  lint-docker:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Lint Dockerfile
        uses: hadolint/hadolint-action@v3.1.0
        with:
          dockerfile: "Dockerfile"
```

In **GitLab CI**:
```yaml
lint_dockerfile:
  stage: test
  image: hadolint/hadolint:latest-debian
  script:
    - hadolint Dockerfile
```

## Check your understanding
1. Why do CI pipelines stop execution when a step exits with a code > 0?
2. Why is linting a `Dockerfile` *before* building it better than analyzing the final built image?

## Going Further
Try configuring Hadolint globally so its output is formatted as JSON by using `hadolint -f json Dockerfile`. This is very useful for Custom UI reporting in modern CI systems!
