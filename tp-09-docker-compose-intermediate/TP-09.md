## Instructions

We'll add a database management tool (Adminer) and explicitly define our volumes and networks.

### Step 1: Examine the Configuration
Open `docker-compose.yml`.

Notice the additions:
- We added a PostgreSQL database (`db`).
- We added `adminer` to manage the database via a web UI.
- We defined **Volumes** (`db-data`) to persist the database.
- We defined **Networks**:
  - `front-tier`: For services exposed to the web (`adminer`).
  - `back-tier`: Internal network for backend and db.

### Step 2: Start the Stack
Start the entire stack in the background:

```bash
docker compose up -d
```

### Step 3: Verify Persistence
Open Adminer at `http://localhost:8080`.
Log in with:
- System: `PostgreSQL`
- Server: `db`
- Username: `postgres`
- Password: `secretpassword`
- Database: `postgres`

Create a table or add some data.

Then, completely destroy the stack and bring it back up:
```bash
docker compose down
docker compose up -d
```
Log back into Adminer. Your data is still there! This is because `docker compose down` does not delete volumes by default.

To delete volumes, you must run: `docker compose down -v`

### Step 4: Network Isolation
We have explicitly put `adminer` and `db` on different networks, but `db` is shared.
Let's see if we can reach Adminer from the database container:

```bash
docker compose exec db ping adminer
```
*Expected output*: `ping: adminer: Name or service not known`

This shows that containers can communicate **only** on the networks they share! `adminer` can talk to `db` because they both share `back-tier`, but `db` cannot talk to other components purely on `front-tier` (if there were any).

Docker Compose makes it easy to horizontally scale services (assuming they don't have conflicting static port mappings!).

### Step 6: Teardown
To cleanly stop all containers and **remove the volumes**:

```bash
docker compose down -v
```
