# TP-05: Dockerfile Best Practices (Hadolint Intro)

**Difficulty**: Intermediate 🟡  
**Estimated Duration**: 30 min  

## Learning Objectives
- Identify common bad practices in Dockerfiles
- Use `hadolint` to lint a Dockerfile locally
- Apply best practices: slim images, non-root users, cleaning package caches

## Prerequisites
- Completed TP-02
- `hadolint` installed locally OR available via Docker

---

## Instructions

We've provided a `Dockerfile.bad` file containing several common anti-patterns and a `Dockerfile.good` file with improvements applied.

### Step 1: Examine the Bad Dockerfile
Look inside `Dockerfile.bad`.
You will notice it uses the huge `ubuntu:latest` image, installs packages without cleaning up the `apt` cache, runs as `root`, and has useless `RUN` layer duplications.

### Step 2: Lint with Hadolint
Let's run `hadolint` against it to see what it says.

If you have it installed locally:
```bash
hadolint Dockerfile.bad
```

If you don't have it installed locally, use the Docker image to run it:
```bash
docker run --rm -i hadolint/hadolint < Dockerfile.bad
```

*Expected output*: You will see multiple warnings!
- `DL3007`: Using latest is prone to errors.
- `DL3008`: Pin versions in apt get install.
- `DL3009`: Delete the apt-get lists after installing something.
- `DL3059`: Multiple consecutive RUN instructions.
- `DL3002`: Last USER should not be root.

### Step 3: Examine the Good Dockerfile
Open `Dockerfile.good`. Notice the changes:
1. `FROM ubuntu:22.04` (Specific tagged version instead of `latest`)
2. `RUN apt-get update && apt-get install -y --no-install-recommends ... && rm -rf /var/lib/apt/lists/*` (Combined layers and cleaned the cache!)
3. Added a `USER nonroot` instruction at the end to avoid running the container as root.

### Step 4: Lint the Good Dockerfile
```bash
hadolint Dockerfile.good
```
No output means the file is clean and respects best practices!

### Step 5: Ignore Rules (Optional)
Sometimes you might want to explicitly ignore a specific rule. We included a `hadolint-config.yaml` file.
Try running the bad file again with the config file that ignores `DL3008` (pinning versions):

```bash
hadolint --config hadolint-config.yaml Dockerfile.bad
```

## Check your understanding
1. Why is deleting `/var/lib/apt/lists/*` important?
2. Why should we combine `apt-get update` and `apt-get install` into a single `RUN` layer?
3. What is the security risk of not using the `USER` directive?
