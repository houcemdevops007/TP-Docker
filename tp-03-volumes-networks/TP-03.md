# TP-03: Volumes and Networks

**Difficulty**: Beginner 🟢  
**Estimated Duration**: 45 min  

## Learning Objectives
- Use named volumes for persistent data.
- Use bind mounts to map host directories into containers.
- Create user-defined bridge networks.
- Connect containers to the same network and observe DNS resolution.

## Prerequisites
- Completed TP-01 and TP-02

---

## Instructions

### Step 1: Named Volumes (Persistent Data)
When a container is deleted, all data inside it is gone. To persist data (like a database or logs), we use volumes.

Let's create a named volume:
```bash
docker volume create my-data-volume
```

Check that the volume exists:
```bash
docker volume ls
docker volume inspect my-data-volume
```

Now, let's run an Alpine container that mounts this volume to `/data`, writes a file into it, and then stops.
```bash
docker run --rm -v my-data-volume:/data alpine sh -c "echo 'Persisted Message!' > /data/message.txt"
```
*Note*: The `--rm` flag removes the container immediately after it finishes running, but the volume is completely separate!

Let's verify the data is still there by creating a completely new container mounting the same volume:
```bash
docker run --rm -v my-data-volume:/data alpine cat /data/message.txt
```
*Expected output*: `Persisted Message!`

### Step 2: Bind Mounts (Sharing the Host directory)
A bind mount maps an exact path on your host OS into the container. This is very useful in development to see local code changes immediately inside the container.

Run an nginx container and map the current directory (`$PWD`) to the default web root (`/usr/share/nginx/html`).
First, create an `index.html` file on your host:
```bash
echo "<h1>Hello from Bind Mount</h1>" > index.html
```

Run the container using a bind mount:
```bash
docker run -d --name nginx-bind -p 8080:80 -v "$(pwd):/usr/share/nginx/html" nginx:alpine
```

Check the result in your browser or with curl:
```bash
curl http://localhost:8080
```

Now, change the HTML on your host machine without restarting the container:
```bash
echo "<h1>Updated directly on the host!</h1>" > index.html
curl http://localhost:8080
```
Notice how fast development can be?

### Step 3: Custom Bridge Networks
Docker containers running on the default `bridge` network cannot resolve each other by name. Only **user-defined** bridge networks provide automatic DNS resolution.

Create a custom network:
```bash
docker network create my-net
docker network ls
```

Start the first container attached to the network and give it a name (`server`):
```bash
docker run -d --name server --network my-net alpine sleep infinity
```

Start another container interactively on the exact same network:
```bash
docker run -it --rm --name client --network my-net alpine sh
```

From inside the `client` container, ping the `server` container by its name:
```bash
/ # ping -c 3 server
/ # exit
```
Notice how the container name `server` resolved to its internal IP address! This DNS resolution allows apps and databases to talk seamlessly without hardcoded IPs.

### Step 4: Cleanup
```bash
docker stop nginx-bind server
docker rm nginx-bind server
docker volume rm my-data-volume
docker network rm my-net
rm index.html
```
