TP-04 : Architecture Web Multi-Conteneurs Manuelle

Intermédiaire ⏱ Durée estimée : 45 min

🎯 Objectifs d'apprentissage

  • Comprendre la nécessité d'orchestrer soi-même une architecture complète à plusieurs briques (Base de données + Backend).
  • Passer de multiples arguments, notamment des variables d'environnement.
  • Comprendre l'extrême complexité asynchrone qui justifie par ailleurs le besoin de Docker Compose (abordé ultérieurement).

📝 Étape par étape

Notre application fournie en exemple dans app/app.py est un petit script Python. Elle se connecte à un serveur Redis et incrémente un compteur de visiteurs globaux.

Étape 1 : Le Réseau Commun

Créons le réseau pour que DB et Application Web s'écoutent :

docker network create app-globale-net

Étape 2 : Démarrage du Cache Redis

On demande à Docker de télécharger l'image de cache ultra-rapide Redis, et on attache ce conteneur à notre réseau.

docker run -d --name serveur-redis --network app-globale-net redis:alpine

Remarque majeure : Pas de directive `-p` ici. On n'expose jamais les bases de données aux hackers (réseau physique), le Backend viendra parler localement à sa base via le réseau isolé !

Étape 3 : Compilation de l'image de l'application

Assurez-vous qu'un Dockerfile est bien présent à la racine de tp-04-simple-webapp, puis compilez :

docker build -t compteur-visite:v1 .

Étape 4 : Lancement du Backend et Configuration d'environnement

Le backend Python ne sait *pas* comment s'appelle la base Redis, le code dit par défaut `localhost`, mais dans Docker, localhost c'est le conteneur python lui-même... Nous allons donc manipuler ce paramètre via le drapeau optionnel d'environnement : -e

docker run -d --name web-api \
    -p 8080:5000 \
    --network app-globale-net \
    -e REDIS_HOST=serveur-redis \
    compteur-visite:v1

Le Backend va tenter de résoudre le domaine serveur-redis. Comme par enchantement, le DNS de Docker lui fournira l'IP interne du cache.

Étape 5 : Test final

Allez sur http://localhost:8080 et bombardez la page de "F5".

Une base Redis stocke toutes les valeurs sous forme de chaîne de caractères ultra rapide.

Si tout fonctionne, vous devriez voir le compteur de hits de l'API s'incrémenter globalement.

Étape 6 : Nettoyage

docker stop web-api serveur-redis
docker rm web-api serveur-redis
docker network rm app-globale-net

❓ Questions de synthèse

  1. Imaginons une architecture classique à 10 micro-services (Auth, Frontend, Data Processing, etc). La commande docker run manuelle avec arguments est-elle facilement viable avec une petite équipe de dix développeurs essayant d'installer ceci la toute première fois ?
  2. Si notre application devait redémarrer suite à une mise à jour Python, que deviendrait la valeur d'incrément des "visites" sur Redis ? (HINT : a-t-on monté un volume nommé dans la base redis lors du TP ?)
