TP-07 : Scan Applicatif et Système avec Trivy

Avancé ⏱ Durée estimée : 30 min

🎯 Objectifs d'apprentissage

  • Exécuter un scanner asynchrone pour déceler passivement les Common Vulnerabilities and Exposures (CVE) dans les librairies O.S d'une image Docker existante.
  • Configurer Trivy de Aqua-Security et filtrer la sévérité ciblée (HIGH/CRITICAL).
  • Empêcher la livraison d'une image vérolée vers un registre d'entreprise de manière dynamique.

📝 Étape par étape

Étape 1 : Fabriquer la victime (Image Vulnérable)

Observez Dockerfile.vulnerable. Ce fichier force le tag pré-historique ubuntu:18.04 et demande volontairement la librairie libssl1.1 de version d'époque (avant des immenses audits de failles de sécurité).

Construisons-la sur le disque et baptisons-la secure-app:vulnerable :

docker build -f Dockerfile.vulnerable -t secure-app:vulnerable .

Étape 2 : Scan Standard via Trivy

Scannons maintenant la création (Il est conseillé de passer par son Docker si Trivy n'est pas nativement installé sur le PC). L'argument du socket docker permet à Trivy de voir et décomposer votre disque image local.

docker run --rm -v /var/run/docker.sock:/var/run/docker.sock aquasec/trivy image secure-app:vulnerable

Observez. Le scanner trouve la distribution (Ubuntu Bionic 18) et compile plus de DEUX-CENT alertes qui défilent. Le tableau montre pour chaque vulnérabilité la référence `CVE`, le packet incriminé, et s'il existait déjà une mise à jour correctrice (`Fixed Version`).

Étape 3 : Filtre de "Tolérance de la Douleur" et Fail du Job

En entreprise, c'est intenable d'interdire le commit et la livraison en production pour la moindre CVE de type "LOW" / "UNKNOWN" sans patch correctif exisant et mineur. La tolérance générale accepte le `Exit 0` sans crash pipeline pour ces petites blessures.

Exécutons notre script pré-préparé :

chmod +x scripts/run-trivy.sh
./scripts/run-trivy.sh secure-app:vulnerable

Script Décortiqué :

  • --severity HIGH,CRITICAL (n'interceptez et filtrez que le grave de chez grave)
  • --ignore-unfixed (ne plantez pas s'il n'y a de toute façon aucun patch publié qui résout le soucis découvert).
  • --exit-code 1 (demander au programme Trivy d'écraser son code de succès final en lui imposant l'échec).

Étape 4 : Validation du fix sur base Alpine Pure

Regardez Dockerfile.fixed. C'est du "Hardened" propre, alpine ultra limitative, sans résidu d'outil complexe.

docker build -f Dockerfile.fixed -t secure-app:fixed .
./scripts/run-trivy.sh secure-app:fixed

Le statut vert absolu est renvoyé ("passed the security scan!") !

Étape 5 : Nettoyages Disque

docker rmi secure-app:vulnerable secure-app:fixed

❓ Questions de validation

  1. Quelle vulnérabilité "énorme" le fait de changer l'O.S. parent (le `FROM ...`) apporte-t-il vis-à-vis d'un package applicatif propre NodeJS ou Python ?
  2. Pourquoi est-il absurde d'exiger d'un pipeline qu'il rejète les "LOW CVE Without FIX" lors d'un build en plein jour de la semaine ?
