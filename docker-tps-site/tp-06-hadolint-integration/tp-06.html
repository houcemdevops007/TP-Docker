TP-06 : Automatisation Check-In (CI) via Script Bash

Intermédiaire ⏱ Durée estimée : 20 min

🎯 Objectifs d'apprentissage

  • Savoir déclencher l'échec ou la réussite d'un pipeline complet en traitant le code erreur (`Exit 1`).
  • Scripter la vérification universelle de tous les Dockerfiles du répertoire.

📝 Étape par étape

Étape 1 : Rôle central du Pipeline Automatise

En entreprise moderne avec GitLab, CircleCI ou GitHub Actions, plus personne ne tape hadolint Dockerfile un par un. Un système valide à chaque Push si votre collègue n'a rien cassé.

Regardez ci-hadolint.sh. Il liste tous les fichiers contenant "*Dockerfile*", tente de lancer la version locale de hadolint et, cas échéant, il lance automatiquement la version par conteneur jetable.

Étape 2 : Simulation locale

Rendez-le exécutable (ce qui reste nécessaire à son passage côté exécuteur distant !).

chmod +x ci-hadolint.sh
./ci-hadolint.sh

Le Dockerfile propre qui se trouve localisé dans ce TP (une toute petite base nginx alpine avec création `appgroup` et `appuser` natif), passera ce check au vert absolu ! (Exit Code de `0`).

Étape 3 : Sabotage

Modifiez en cassant explicitement notre Dockerfile du répertoire : rajoutez-y une directive FROM node:latest.

Ré-exécutez le script :

./ci-hadolint.sh

Il vous interceptera ("Linting... DL3007") et plantera directement en sortant sans exécuter la ligne finale de bravoure !

Étape 4 : Démonstration sur GitHub Action et GitLab

Pour culture DevOps complète, comment cela se passe-t-il sur GitLab CI par exemple ? Très simple : 

lint_code_docker:
  stage: test_statique
  image: hadolint/hadolint:latest-debian
  script:
    - hadolint Dockerfile
