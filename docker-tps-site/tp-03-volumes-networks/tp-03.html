TP-03 : Volumes et Réseaux (Networks)

Débutant ⏱ Durée estimée : 45 min

🎯 Objectifs d'apprentissage

  • Utiliser des volumes nommés pour assurer la persistance de l'état (bases de données).
  • Utiliser les bind mounts pour brancher un dossier local dans Docker (idéal dev local).
  • Créer un réseau de type bridge défini par l'utilisateur.
  • Vérifier la bonne résolution DNS inter-conteneurs (communication via le nom et non l'IP).

📝 Étape par étape

Étape 1 : Les Volumes Nommés (Persistance Absolue)

Si vous détruisez un conteneur MySQL, les données meurent avec lui. Pour éviter cela, Docker gère des disques durs virtuels appelés "Volumes".

Créons un volume manuellement :

docker volume create mon-stockage
docker volume ls

Lançons un petit conteneur qui écrit un mot de passe dans ce fameux volume localisé dans le dossier /data, puis qui se détruit immédiatement (--rm) :

docker run --rm -v mon-stockage:/data alpine sh -c "echo 'MotDePasseSecret123' > /data/fic.txt"

Vérifions maintenant si la donnée a survécu à la suppression du conteneur en lançant un autre conteneur complétement différent qui "branche" (monte) le même disque :

docker run --rm -v mon-stockage:/data alpine cat /data/fic.txt

Résultat magique : la chaîne "MotDePasseSecret123" est toujours là. Le cycle de vie des données a été découplé du cycle de vie du conteneur !

Étape 2 : Les Bind Mounts (Partage de dossier Hôte-Conteneur)

Un cas classique pour un développeur consiste à vouloir que le code source exécuté par nginx ou nodeJS reflète instantanément le fichier modifié sur la machine physique via VSCode. C'est le but du Bind Mount.

Commencez par créer un fichier sur votre bureau :

echo "<h1>Index du developpeur local</h1>" > index.dev.html

Lançons Nginx en liant notre répertoire courant matériel au dossier virtuel de web racine d'Nginx.
(Sous Windows Powershell remplacez $(pwd) par ${PWD})

docker run -d --name nginx-dev -p 8080:80 -v "$(pwd):/usr/share/nginx/html" nginx:alpine

Visitez http://localhost:8080/index.dev.html. Maintenant, ouvrez le fichier avec le Bloc-notes / VSCode sur votre PC, modifiez le texte, sauvegardez et rafraîchissez simplement la page : C'est instantané !

Étape 3 : Réseaux virtuels internes isolés

Les applications doivent communiquer entre elles (un backend parle à une base de données). Si on les lance sur le port par défaut de Docker, elles ne peuvent de s'adresser mutuellement que par des adresses IP qui changent tout le temps. Solution : créer son propre réseau virtuel.

docker network create mon-reseau-prive

Démarrons le serveur qui s'exécutera à l'infini sur ce réseau :

docker run -d --name serveur-critique --network mon-reseau-prive alpine sleep infinity

Démarrons le client interactif sur ce même réseau :

docker run -it --rm --name client --network mon-reseau-prive alpine sh

À l'intérieur du conteneur client, tentez un PING sur le NOM du serveur (et non son IP) :

/ # ping -c 3 serveur-critique
/ # exit

Une véritable résolution DNS interne est intégrée par les réseaux "bridge" personnalisés !

Étape 4 : Nettoyage

docker stop nginx-dev serveur-critique
docker rm nginx-dev serveur-critique
docker volume rm mon-stockage
docker network rm mon-reseau-prive
rm index.dev.html
