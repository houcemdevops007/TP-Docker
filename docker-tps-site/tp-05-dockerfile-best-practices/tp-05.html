TP-05 : Bonnes Pratiques Dockerfile avec Hadolint

Intermédiaire ⏱ Durée estimée : 30 min

🎯 Objectifs d'apprentissage

  • Identifier les "anti-patterns" (mauvaises pratiques classiques) dans l'écriture d'un Dockerfile.
  • Utiliser le linter de référence hadolint pour scanner votre fichier localement.
  • Appliquer les correctifs pour obtenir une image légère, dé-cachée proprement et non gérée par le super-utilisateur "root".

📝 Étape par étape

Deux fichiers sont fournis à la racine du TP-05 : Dockerfile.bad et un Dockerfile.good corrigé.

Étape 1 : Analyser les aberrations de 'Dockerfile.bad'

Ouvrez le premier fichier. Les pires horreurs possibles s'y trouvent :

  • Utilisation du lourd et dangereux tag ubuntu:latest. (L'OS passera en version suivante dès publication d'Ubuntu d'une nouvelle sous-version ! L'image finale casserait les dépendances Python du jour au lendemain).
  • Multiplication intempestive de la directive RUN (apt-get update tourne dans son coin, stocke du fichier temporaire sur un layer complet de l'historique et apt-get install recrée d'autres layers).
  • Absence totale de directive USER. Si le hacker perçe votre process python sur le port web, il atterrira directement en ROOT sur la base hôte de l'OS partagé !

Étape 2 : Analyser l'échec avec Hadolint

Si vous avez Hadolint localement :

hadolint Dockerfile.bad

Sinon, un conteneur d'Hadolint est utilisable à la volée !

docker run --rm -i hadolint/hadolint < Dockerfile.bad

Hadolint vous hurlera les recommandations suivantes liées aux DL (Docker Lints) :

  • DL3007: L'usage exclusif du tag ":latest" est proscrit. Préferez forcer "ubuntu:20.04" par exemple.
  • DL3008: Épingler strictement la version python3 dans les apt-get. Ex "python3=3.10.*".
  • DL3009/15: Obligation impérative de taper rm -rf /var/lib/apt/lists/* en toute fin de chaîne d'installation pour limiter le poids !
  • DL3059: Trop de RUN consécutifs, on utilise un enchaînement : `RUN apt update && apt install ...`
  • DL3002: Toujours clore un Dockerfile non vide avec USER appuser par exemple.

Étape 3 : La version saine (Good)

Maintenant, analysez Dockerfile.good. Le layer compilé contient un `RUN` unique majestueux et il ajoute la prévention `useradd`.

hadolint Dockerfile.good

Alerte muette ? Fichier validé à 100%.

Étape 4 : Fichier de dérogation externe

Parfois (rarement), une exception subsiste. Lisez hadolint-config.yaml. On peut demander à ignorer via un simple flag :

hadolint --config hadolint-config.yaml Dockerfile.bad

❓ Questions de validation

  1. Quel danger de production l'utilisation de `debian:latest` au point `FROM` présente-t-elle le vendredi soir à 18h en cas de re-build CI automatique ?
  2. De quoi est composée la différence matérielle (poids en Megaoctets téléchargés) entre les images respectives au DL3009 ?
  3. Quelle vulnérabilité le linter vous permet-il d'éviter si l'on ne crée pas d'utilisateur linux bas niveau ?
