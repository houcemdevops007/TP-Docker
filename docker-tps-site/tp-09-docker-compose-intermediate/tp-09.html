TP-09 : Volumes, Multi-Réseaux et Scalabilité

Intermédiaire ⏱ Durée estimée : 40 min

🎯 Objectifs d'apprentissage

  • Isoler strictement ses conteneurs via réseaux virtuels explicitement "Front-Tier" / "Back-Tier" au sein du YAML.
  • Déclarer des volumes nommés persistants dans un contexte Compose.
  • Démontrer la scalabilité ("Mise à l'échelle") horizontale.

📝 Étape par étape

Étape 1 : Le YML De Séparation Physique

Ouvrez et étudiez le docker-compose.yml de ce module. Nous intégrons un PostgreSQL et un gestionnaire visuel `Adminer` pour naviguer dans la base.

Notez la nouvelle nomenclature au premier niveau du YAML (les racines `services:`, `networks:` et `volumes:` cohabitent). C'est le bloc de déclaration initial.

Étape 2 : Lancement

docker compose up -d

Étape 3 : Persistance Implicite de Compose

Rendez-vous sur `http://localhost:8080/`. Identifiez-vous (PostgreSQL / Serveur: db / user: postgres / pass: secretpassword). Créez vite fait une bdd `test`.

Détruisons l'écosystème puis relançons-le pour simuler une énorme mise à jour : 

docker compose down
docker compose up -d

Magie ! Vos tables sont conservées. Par mesure de fiabilité, un simple `compose down` conserve le volume db-data intact rattaché au nom du projet !

Étape 4 : Preuve d'Isolation Réseau Absolue

Notre `db` écoute dans l'ombre sur `back-tier`. Adminer, public, connecte les mondes `front-tier` et `back-tier`.

Faisons pinger le `adminer` DEPUIS la base `db` (elles partagent un réseau interne donc tout va bien).

docker compose exec db ping adminer

Imaginons un service 3 (ex: VueJS) branché purement sur le Front. Jamais aucune ligne de code ne pourrait deviner l'IP ou le nom de `db` car l'isolation des réseaux Docker est totale côté DNS.

Étape 5 : Nettoyage Complet (Destruction des données)

Il est temps de purger, disques SQL y compris. Remarquez le `-v` à la fin (signifiant: purgez aussi les Volumes déclarés !) :

docker compose down -v

❓ Questions de validation

  1. Un développeur fait un `docker compose down` pour rentrer chez lui à 18h. Le lendemain, la donnée SQL modifiée à 17h59 sera-t-elle présente ?
  2. Pourquoi brancher une API à un réseau frontal mais la BDD uniquement au réseau arrière ?
