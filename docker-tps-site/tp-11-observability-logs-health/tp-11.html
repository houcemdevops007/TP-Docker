TP-11 : Drivers de Logs Limitatifs et Sonde HTTP Applicative

Avancé ⏱ Durée estimée : 30 min

🎯 Objectifs d'apprentissage

  • Empêcher l'asphyxie redoutée du "Disk Full" sur le serveur Docker de production (Rotation de logs locaux intégrée).
  • Demander au client HTTP WGET (Curl proxyfié) minimal d'interroger logiciellement le statut en interne au code.

📝 Étape par étape

Étape 1 : Le Cauchemar du fichier de LOG Infini

Ouvrez le fichier local docker-compose.yml. Par instinct un codeur laisse l'app loguer par nature, mais Docker agglomère ça indéfiniment. Regardez la partie logging: driver: json-file.

Ce bloc salvateur écrète magiquement les journaux locaux passés les 3 roulements de 10 MegaOctets ! (30 Mo absolus conservés maximum sans polluer l'infrastructure du cloud).

Étape 2 : Coder un faux chargement Python lent

Dans app/app.py, constatez la petite tricherie ajoutée sous la forme de l'endpoint d'API web `/health`. Le code ne répondra délibérement avec l'OK HTTPS que passé 10 secondes pleines (afin de forcer un temps de chauffe applicatif visible humainement simulé).

Étape 3 : Build & Observation du temps de chauffe asynchrone

docker compose up -d --build
watch -n 1 "docker compose ps"

(Pour quitter watch : tapez "Crtl + C").

Durant ses 10 premières secondes pénibles, vos requêtes wget automatiques depuis l'intérieur échouent. Le conteneur reste grisé `(health: starting)`, puis il clignotte sur la consigne Verte Libératrice !

Étape 4 : Log Tagging Central

docker compose logs -f api

Compose marque explicitement la ligne lue et l'identifiant pour la centralisation simplifiée. Testez de bombarder en local via `curl http://localhost:8080/health` pendant que votre terminal 1 regarde tourner infiniment les logs, générés à la réactivité micro-secondes.
