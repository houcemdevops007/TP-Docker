TP-10 : Sécurisation, Limitations et Healthchecks Logiques

Avancé ⏱ Durée estimée : 45 min

🎯 Objectifs d'apprentissage

  • Définir et utiliser intelligemment les Sondes de Vie (Healthchecks).
  • Verouiller conditionnellement le démarrage du backend : on démarre UNIQUEMENT quand Postgresql est prête et réceptive, et non simplement "Démarrée".
  • Contraindre physiquement le CPU et la RAM (OOM Killing).

📝 Étape par étape

Étape 1 : Analyser le danger du "Fake Running"

Ouvrez `docker-compose.yml`. Un container lourd (ex: ElasticSearch, MySQL) affiche l'état "Running" dès que son conteneur `docker run` est passé, mais le boot de traitement base de données en interne au processus peut durer 7 à 10 secondes.
Si l'API démarre en même temps et que son driver ne gère pas l'attente... Crash serveur d'API dés la seconde 1 !

C'est ici qu'interviennent les fameux `healthcheck:` et la condition d'attente condition: service_healthy.

Étape 2 : Lancer et Monitorer les Etats

Démarrez la suite :

docker compose up -d
docker compose ps

Re-tapez les PS. Observez l'état du Postgres, il affiche très brièvement (health: starting) puis bascule en (healthy). Le conteneur "api" (simulé par Alpine pour s'endormir à l'infini ici) ne démarre QUE passé ce délai sain !

Étape 3 : Le Plafond De Verre (Cgroups)

La limite mémoire de JVM ou de Node est incontournable.

docker stats

Validez vos limites : vous devriez voir la ligne "api" bridée de force (MEM LIMIT) à 128 MB maximum. Si elle dépassait (memory leak script), Docker la fusillerait via un OOM-Kill instantané (Restart par défaut) au lieu de laisser crasher toute la machine hôte linux serveur principal.

Étape 4 : Démonstration de la condition de garde

Éditez le fichier YAML local pour simuler une panne SQL grave. Remplacer `-U postgres` par un compte qui n'existe pas : `-U COMPTE_INVALID_CRASH` de la ligne de test du composant DB.

Relancez pour observer le désastre sous contrôle absolu :

docker compose down
docker compose up -d
docker compose ps

La DB sera incapable de passer son tir de check de vie, elle s'affichera (unhealthy). L'API d'entreprise sera complètement empêchée et bloquée en statut "Starting" par Docker pour protéger l'amorçage. Incide évité.

❓ Questions de validation
