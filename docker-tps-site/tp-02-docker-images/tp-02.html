TP-02 : Construire ses premières images (Dockerfile)

Débutant ⏱ Durée estimée : 45 min

🎯 Objectifs d'apprentissage

  • Maîtriser les instructions de base d'un Dockerfile : FROM, RUN, COPY, CMD, EXPOSE.
  • Fabriquer une image Docker personnalisée à partir du code source d'une application.
  • Appliquer un tag personnalisé (-t) à l'image et l'exécuter.

🛠 Prérequis

  • Avoir terminé le TP-01 avec succès.

📝 Étape par étape

Pour cet exercice, une petite application web codée en Python (Flask) est fournie dans le dossier app/ à côté du README.

Étape 1 : Comprendre l'application source

Ouvrez et regardez le contenu de app/app.py et de app/requirements.txt (le fichier qui liste les dépendances python nécessaires). Cette application web doit écouter sur le port 5000.

Étape 2 : Rédiger le Dockerfile

A la racine du dossier du TP-02 (là où se trouve le dossier app/), créez un fichier nommé exactement Dockerfile (sans aucune extension). Remplissez le avec le contenu suivant :

# 1. On part d'une image officielle Python (version allégée "slim")
FROM python:3.9-slim

# 2. On définit le dossier de travail PAR DÉFAUT à l'intérieur du conteneur
WORKDIR /usr/src/app

# 3. OPTIMISATION : On copie SEULEMENT le fichier des dépendances dans un premier temps
COPY app/requirements.txt ./

# 4. On installe les librairies Python demandées
RUN pip install --no-cache-dir -r requirements.txt

# 5. On copie le reste du code de notre projet
COPY app/ ./

# 6. Documentation : On indique que ce conteneur a vocation à écouter sur le port 5000
EXPOSE 5000

# 7. La commande exécutée par défaut au démarrage du conteneur
CMD ["python", "app.py"]

Explication cruciale : Pourquoi on ne copie pas tout le dossier app/ d'un seul coup (ligne 3) ? Parce que Docker met en cache chaque ligne. L'installation des librairies Python (pip install) peut être longue. Si on copie app/app.py (qui change souvent) avant l'installation des dépendances, Docker referait l'installation de zéro à chaque petite modification de code !

Étape 3 : Compiler (Build) l'image

Maintenant, on demande à Docker de lire le fichier et de générer l'image :

docker build -t hello-devops:v1 .

Astuce : Le point . à la toute fin n'est pas une faute de frappe, il indique à Docker que le dossier contenant le projet (le "contexte") est le dossier actuel.

Vérifiez que l'image existe bien :

docker images | grep hello-devops

Étape 4 : Lancer le conteneur maison

C'est le moment de tester notre création en liant notre port local 8080 vers le port 5000 de l'application :

docker run -d --name mon-app-python -p 8080:5000 hello-devops:v1

Vérifiez que le serveur répond correctement :

curl http://localhost:8080

Vous devriez voir un résultat JSON : {"message":"Hello DevOps ver. 1.0"}.

Étape 5 : Modification de code et re-compilation ultra-rapide

Modifiez le fichier app/app.py sur votre machine pour renvoyer le texte : "Hello DevOps (V2)".

Relancez ensuite la compilation en changeant la version textuelle du tag :

docker build -t hello-devops:v2 .

Avez-vous remarqué à quel point l'étape d'installation avec pip a été passée instantanément ? C'est le fameux Cache Docker !

Étape 6 : Nettoyage

docker stop mon-app-python
docker rm mon-app-python

❓ Questions de validation

  1. À quoi sert l'instruction WORKDIR dans le Dockerfile ?
  2. Pourquoi est-il indispensable de séparer la copie du requirements.txt et la copie de app.py ?
  3. Que signifie le . (point) à la fin de la commande docker build -t nom:v1 . ?

🚀 Pour aller plus loin...

Ouvrez un dossier vide et créez votre propre Dockerfile qui utilise l'image nginx:alpine. Écrivez un fichier index.html simple en local, et utilisez la commande COPY pour l'injecter dans /usr/share/nginx/html/index.html. Compilez et exécutez votre serveur web 100% statique personnalisé !
