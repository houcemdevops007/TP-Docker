# TP-10: Docker Compose Advanced

**Difficulty**: Advanced 🔴  
**Estimated Duration**: 45 min  

## Learning Objectives
- Use Healthchecks to ensure services are actually ready, not just running
- Understand `depends_on` with `condition: service_healthy`
- Set CPU and Memory resource limits on containers

## Prerequisites
- Completed TP-09

---

## Instructions

A container might be "running", but the database inside it might take 10 seconds to initialize. If your web app starts immediately, it will crash because it can't connect to the DB. Healthchecks solve this.

### Step 1: Examine the Configuration
Open `docker-compose.yml`.

Look at the `db` service. It has a `healthcheck` block:
```yaml
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U postgres"]
  interval: 5s
  timeout: 5s
  retries: 5
```

Look at the `api` service. It depends on `db`, but specifically waits for it to be healthy:
```yaml
depends_on:
  db:
    condition: service_healthy
```

Also, notice the `deploy.resources` block for the `api` service. It limits the container to 0.5 CPU cores and 128MB of RAM.

### Step 2: Start the Stack and Observe
Start the stack and immediately check the processes:

```bash
docker compose up -d
docker compose ps
```
You should see the `db` state as `(health: starting)`. 
Wait a few seconds and run `docker compose ps` again. It will change to `(healthy)`, and only *then* will the `api` container start!

### Step 3: View Container Stats
Let's see the resource limits in action.

```bash
docker stats
```
Look at the `MEM LIMIT` column for the `api` container. It should be capped at `128MiB`.
Press `Ctrl+C` to exit.

### Step 4: Break the Healthcheck
Edit the `docker-compose.yml` file and change the healthcheck command to something broken:
`test: ["CMD-SHELL", "pg_isready -U WRONGUSER"]`

Bring the stack down and back up:
```bash
docker compose down
docker compose up -d
docker compose ps
```
Watch what happens. The `db` will become `(unhealthy)`, and the `api` will never start because its condition (`service_healthy`) was never met. This prevents cascading failures in production!

### Step 5: Teardown
Restore the correct username in the YAML file, then clean up:
```bash
docker compose down -v
```
